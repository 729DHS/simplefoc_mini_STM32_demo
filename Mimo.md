根据提供的参考内容，这是一张**嵌入式系统（特别是STM32微控制器）项目构建后生成的内存分布映射表**。它详细展示了程序编译链接后，代码和数据在物理内存（RAM和FLASH）中的具体存储地址和占用大小。

### 这是什么？
这是项目链接器（Linker）生成的一个**内存布局报告**。它将程序中的不同组成部分（如代码、变量、常量）分配到微控制器的实际内存物理地址上。

表格中包含两个主要区域（region）：
1.  **RAM region**：起始于地址 `0x20000000`，总大小为 20480 字节。
    *   `data.` 段：存放已初始化的全局变量和静态变量。
    *   `bss.` 段：存放未初始化或初始化为0的全局变量和静态变量。
    *   `user_heap_stack.` 段：为堆（动态内存分配）和栈（函数调用、局部变量）预留的空间。

2.  **FLASH region**：起始于地址 `0x08000000`，总大小为 65536 字节。
    *   `isr_vector.` 段：存放中断向量表。
    *   `text.` 段：存放程序的可执行代码。
    *   `rodata.` 段：存放只读数据（如常量、字符串）。
    *   `ARM.` 段：存放ARM架构特定的异常处理信息。
    *   `init_array.` 和 `fini_array.` 段：存放程序启动和退出时需要调用的函数指针。

### 跟课本上学的STM32内存区域有什么关系？
这与你在课本上学到的STM32内存区域划分**完全对应，是课本理论的具体实现和体现**。

1.  **地址的对应关系**：
    *   课本中提到的STM32的FLASH存储器，其起始地址通常就是 `0x08000000`，这与表格中`FLASH` region的起始地址完全一致。
    *   课本中提到的STM32的内置SRAM（RAM），其起始地址通常就是 `0x20000000`，这也与表格中`RAM` region的起始地址完全一致。

2.  **功能区域的对应关系**：
    *   **课本知识**：会告诉你FLASH用于存储程序代码和常量，RAM用于存储运行中的变量、堆栈等。
    *   **表格体现**：这张表精确地展示了这些功能是如何实现的：
        *   程序代码（`text.`）和常量（`rodata.`）被链接到了FLASH地址空间（`0x0800xxxx`）。
        *   变量（`data.` 和 `bss.`）以及堆栈空间（`user_heap_stack.`）被链接到了RAM地址空间（`0x2000xxxx`）。

**简单来说，你课本上学的是STM32芯片“有哪些内存区，以及它们是干什么用的”（理论）。而这张表，是你的具体项目代码“在这些内存区里具体怎么放、放在哪里”（实践）。** 它验证并直观地展示了课本所描述的内存模型。



# STM32F103 I2C 卡死问题分析与解决思路

## 问题本质

你的代码注释中已经准确描述了这个问题：

> "STM32F1 I2C 外设有时会卡在 BUSY 状态，此时 HAL_I2C_Mem_Read 不报错但返回旧值"

这是 **STM32F1 系列 I2C 硬件层面的已知缺陷**，核心在于：
- I2C 总线在特定信号时序下进入 BUSY 状态无法自恢复
- HAL 库的超时机制不够完善，有时无法正确检测到挂起状态
- 中断驱动模式下更容易触发此问题

## 你当前代码中的防护措施

从代码中可以看到，你已经做了两层防护：

| 防护机制 | 检测条件 | 动作 |
|---------|---------|------|
| **I2C 卡死检测** | 连续 ≥100 次读值不变（约 0.9 秒） | 硬件复位 I2C1 外设 |
| **I2C 通信失败** | `HAL_I2C_Mem_Read` 返回非 `HAL_OK`，连续 ≥60 次 | 禁用电机，PWM 置中位 |

硬件复位的方式是：
```c
__HAL_RCC_I2C1_FORCE_RESET();
__HAL_RCC_I2C1_RELEASE_RESET();
HAL_I2C_Init(&hi2c1);
```

## 你代码中可能存在的问题

### 1. 卡死检测阈值太高，检测到时已经"太晚"

连续 100 次读值不变才触发复位，按 ~110Hz 的读取频率算，大约要 **0.9 秒** 才能检测到。在这 0.9 秒内，系统一直在用**旧的编码器角度**做 PID 计算和 PWM 输出，电机行为会非常异常。代码中也说了"值变了→I2C 正常"，但如果 I2C 一卡就返回旧值，这 0.9 秒内 PID 积分项会持续累积，导致复位后电机剧烈抖动。

### 2. 卡死复位后 `continue` 跳过了 PID 更新

```c
if (g_i2c_stuck_cnt >= 100) {
    // ... 复位 I2C ...
    continue;   // 跳过本轮 PID + PWM 更新
}
```

复位后跳过了本轮控制更新，但**没有重置 PID 的积分项**。旧积分值可能已经累积到很大的值，复位后第一帧有效数据进来时 PID 输出会突变。

### 3. I2C 没有做 SCL/SDA 引脚级的恢复

你用的是 RCC 级别的外设复位，这只能复位 I2C 外设的寄存器状态。但如果是**总线上 SDA 被从设备拉低不释放**（AS5600 侧异常），仅复位 I2C 外设是不够的，需要手动操作 GPIO 发送时钟脉冲把 SDA 释放出来。

### 4. 定时器中断与 I2C 读取的时序问题

从代码看，`foc_tick` 由 TIM2 中断设置（~1098Hz），主循环轮询这个标志。I2C 读取和 PID 计算都发生在主循环中。`HAL_I2C_Mem_Read` 是阻塞调用（超时 2ms），如果 I2C 处于异常状态，这个阻塞等待会拖慢主循环，导致 `foc_tick` 堆积或者丢失实时性。

## 其他使用者的常见解决方案

使用 STM32F103 + AS5600 做 FOC 控制的方案并不少见，社区中有几种不同的应对策略：

### 方案一：用模拟 I2C（软件 I2C）替代硬件 I2C

这是**最常见的"终极解决方案"**。HAL 硬件 I2C 的 bug 太多太深，很多开发者干脆放弃硬件 I2C，用 GPIO 模拟 I2C 时序。模拟 I2C 完全不依赖 I2C 外设寄存器状态，不存在 BUSY 卡死的问题。

- 优点：彻底避开硬件 bug，可靠性大幅提升
- 缺点：需要 CPU 参与时序生成，会占用一些主循环时间；速度比硬件 I2C 略慢

### 方案二：换用 SPI 接口的编码器

AS5600 是 I2C 接口的，但同系列有 SPI 接口的型号（如 **AS5048A/AS5047P**），或者用 **TLE5012B** 等 SPI 磁编码器。STM32F103 的 SPI 外设没有类似的已知严重 bug，且 SPI 速度更快、时序更简单。

### 方案三：使用 AS5600 的 PWM/模拟输出模式

AS5600 除了 I2C，还支持 **PWM 输出** 和 **模拟电压输出**。可以用 STM32 的 ADC（模拟模式）或输入捕获（PWM 模式）来读取角度，完全绕开 I2C：
- 模拟输出：接 ADC，读取电压换算角度，速度最快、最简单
- PWM 输出：用定时器输入捕获，精度略低但不依赖 I2C

### 方案四：更激进的 I2C 恢复策略

如果坚持用硬件 I2C，需要实现**引脚级别的总线恢复**：
1. 检测到卡死后，把 I2C 的 SCL/SDA 引脚临时切换为 GPIO 输出模式
2. 手动发送最多 9 个 SCL 时钟脉冲，直到 SDA 被释放
3. 产生一个 STOP 条件
4. 再重新初始化 I2C 外设

这比你当前的 RCC 复位要彻底得多。

### 方案五：改用 DMA 或中断方式读取 I2C

你的代码用的是阻塞式 `HAL_I2C_Mem_Read`，可以尝试改为 `HAL_I2C_Mem_Read_IT` 或 `HAL_I2C_Mem_Read_DMA`，配合状态机在主循环中非阻塞地完成读取，避免一个卡死的 I2C 调用阻塞整个控制循环。

## 针对你当前代码的改进建议

如果暂时不想大改架构，可以先做以下调整：

1. **降低卡死检测阈值**：从 100 次改为 20~30 次（约 0.2 秒），更快触发恢复
2. **复位后重置 PID**：在 `continue` 之前调用 `PID_Reset(&motor.pid)`，避免旧积分导致突变
3. **复位后清空 `g_mech_angle_invalid` 标志**：复位后的第一帧数据应该跳过 PID，仅用来刷新角度基准
4. **增加 SDA 引脚级恢复**：在 RCC 复位之前，先尝试 GPIO 级别的总线恢复
5. **考虑替换为模拟 I2C**：这是最稳妥的长期方案

总结：你遇到的问题不是你的代码逻辑错误，而是 **STM32F103 硬件 I2C 外设的已知缺陷**。代码中已有的防护机制方向是对的，但检测太慢、恢复不够彻底。社区中最主流的做法是**换模拟 I2C**或**换 SPI 编码器**来彻底绕开这个问题。



# 栈溢出的完整机理与正确的串口通信做法

## 栈溢出到底是怎么发生的

你的困惑在于"栈溢出"这四个字听起来很抽象。让我用你代码里的真实场景一步步推演：

### 栈是怎么工作的

栈是从高地址向低地址增长的。你代码中每次函数调用，局部变量都会"压"到栈上，函数返回时再"弹"出来。但问题是——**如果函数还没返回，又被新的函数调用压了更多东西上去，栈就会越来越深。**

### 你代码中的一次"致命时刻"

假设某一刻，自动打印被触发了，主循环开始执行：

```
调用链（从外到内）：
  main() 的 while 循环                          ← 栈上已有 main 的局部变量
    → UART_PrintStatus()                         ← 压入 384 字节的 char buf[384]
      → snprintf(buf, 384, "...%.1f...", ...)    ← newlib 的 snprintf 内部再压 300~500 字节
                                                   （浮点格式化需要大量中间变量）
        → 此时栈深度已经达到极限，几乎是满的
        → 突然！TIM2 中断来了！
          → HAL_TIM_PeriodElapsedCallback()       ← 又压入几十字节
            → 栈指针越过了 1536 字节的边界
              → 踩到了全局变量 motor, hi2c1 ...
```

**关键点：不是 ticks 越来越大导致溢出，而是 `snprintf` 浮点格式化本身就已经把栈用到了极限，TIM2 中断在这时打断它，就成了压死骆驼的最后一根稻草。**

### 为什么"到一定时间就暂停"

这很好解释：

- 自动打印每 **~0.5 秒** 触发一次
- 每次触发就执行一次 `snprintf` + 浮点格式化
- 每次都踩踏全局变量
- 踩到 `hi2c1` 的内部状态 → I2C 通讯异常
- 踩到 `motor.enabled` 或 `motor.pid` → 控制逻辑混乱
- 多次踩踏积累后，系统行为完全不可预测，表现为"卡死"

**所以看起来是"到一定时间就暂停"，本质是每 0.5 秒被破坏一次，几次之后就彻底崩了。**

## `snprintf` vs 自己写串口发送

这两个东西其实不在同一个层面上：

```
snprintf / printf  →  格式化（把数字变成字符串）   ← 栈溢出的元凶
HAL_UART_Transmit  →  发送（把字符串通过串口发出）  ← 只是慢，不导致溢出
```

`snprintf` 带 `%f` 的问题不是"写串口"的问题，而是**格式化浮点数**这个动作本身需要大量临时内存。你自己写串口发送函数，只要格式化那一步还是用 `snprintf(... "%f" ...)`，问题依然存在。

所以要解决的是**格式化方式**，不是发送方式。

## 正确的做法

### 第一步：替换浮点格式化（立即可做，效果最大）

```c
// ❌ 原来的写法：snprintf 内部会为每个 %f 分配大量栈空间
snprintf(buf, sizeof(buf), "deg:%.1f err:%.1f tgt:%.1f v:%.2f\r\n",
         (double)deg, (double)err, (double)tgt, (double)v);

// ✅ 改为整数近似：乘100后当整数打印，snprintf 不触发浮点代码
int deg_i  = (int)(deg * 100.0f);   // "12.34°" → 1234
int err_i  = (int)(err * 100.0f);
int tgt_i  = (int)(tgt * 100.0f);
int v_i    = (int)(v * 100.0f);

snprintf(buf, sizeof(buf), "deg:%d.%02d err:%d.%02d tgt:%d.%02d v:%d.%02d\r\n",
         deg_i/100, abs(deg_i%100),
         err_i/100, abs(err_i%100),
         tgt_i/100, abs(tgt_i%100),
         v_i/100,   abs(v_i%100));
```

这样 `snprintf` 内部只走整数路径，**不链接浮点格式化库**，栈消耗从几百字节降到几十字节。

### 第二步：把打印缓冲区移到全局（进一步释放栈空间）

```c
// ❌ 栈上分配
static void UART_PrintStatus(void) {
    char buf[384];          // 384 字节全压在栈上
    snprintf(buf, ...);
    UART_SendString(buf);
}

// ✅ 改为全局静态分配（占 BSS 段，不占栈）
static char g_print_buf[128];   // 放在文件作用域，全局区

static void UART_PrintStatus(void) {
    snprintf(g_print_buf, sizeof(g_print_buf), ...);
    UART_SendString(g_print_buf);
}
```

### 第三步：使用 DMA 发送，主循环不阻塞

```c
// 当前写法：阻塞，CPU 干等 17ms 发完
HAL_UART_Transmit(&huart1, (uint8_t *)str, strlen(str), 100);

// 改为 DMA 方式：把数据交给 DMA，CPU 立即返回继续做控制
static volatile uint8_t tx_busy = 0;
static char tx_buf[256];

void UART_SendString_NonBlock(const char *str) {
    if (tx_busy) return;           // 上一次还没发完就丢弃
    uint16_t len = strlen(str);
    if (len > sizeof(tx_buf) - 1) len = sizeof(tx_buf) - 1;
    memcpy(tx_buf, str, len);
    tx_busy = 1;
    HAL_UART_Transmit_DMA(&huart1, (uint8_t *)tx_buf, len);
}

// DMA 发送完成回调
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        tx_busy = 0;
    }
}
```

### 第四步：整体架构——控制与通信分离

正确的嵌入式架构应该是这样的：

```
┌─────────────────────────────────────────────────────┐
│                    主循环                            │
│                                                     │
│  ┌──────────────────┐    ┌───────────────────────┐  │
│  │  高优先级任务     │    │  低优先级任务          │  │
│  │  (每 ~9ms)       │    │  (空闲时执行)          │  │
│  │                  │    │                       │  │
│  │  ① 读编码器      │    │  ④ 检查串口命令       │  │
│  │  ② PID 计算      │    │  ⑤ 格式化状态字符串   │  │
│  │  ③ 更新 PWM      │    │  ⑥ DMA 发送          │  │
│  └──────────────────┘    └───────────────────────┘  │
│                                                     │
│  通信永远不能阻塞控制                                │
└─────────────────────────────────────────────────────┘
```

### 第五步：用环形缓冲区实现命令队列

当上位机频繁发送命令（调 PID、改模式等）时，正确的做法是用环形缓冲区：

```c
// 简单的环形缓冲区
#define CMD_BUF_SIZE 4
static char cmd_queue[CMD_BUF_SIZE][64];
static volatile uint8_t cmd_head = 0;  // 写入位置
static volatile uint8_t cmd_tail = 0;  // 读取位置

// 中断中只做：存入缓冲区
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    // ... 接收逻辑 ...
    // 收到完整命令后：
    if (uart_cmd_ready) {
        uint8_t next = (cmd_head + 1) % CMD_BUF_SIZE;
        if (next != cmd_tail) {           // 队列没满
            memcpy(cmd_queue[cmd_head], uart_rx_buf, 64);
            cmd_head = next;
        }
        uart_cmd_ready = 0;
    }
}

// 主循环中：取出来处理
while (cmd_tail != cmd_head) {
    UART_ParseCommand(cmd_queue[cmd_tail]);
    cmd_tail = (cmd_tail + 1) % CMD_BUF_SIZE;
}
```

## 总结：教训清单

| 教训 | 说明 |
|------|------|
| **嵌入式中尽量不用 `%f` 浮点格式化** | 栈消耗巨大，用整数近似替代 |
| **大缓冲区放全局，别放栈** | `char buf[大数字]` 放函数里就是栈溢出隐患 |
| **串口发送不要阻塞** | 用 DMA，CPU 干完活就走 |
| **通信不能影响实时控制** | 控制是主线，打印是附加，绝不能让打印打断控制节拍 |
| **中断里的代码要尽可能短** | 中断中只设标志 + 存数据，处理放主循环 |
| **上线前估算栈使用量** | CubeIDE 编译后看 `.map` 文件，或者在启动文件里填栈哨兵值检测 |

核心原则就一句话：**在资源受限的 MCU 上，每一个字节的栈和每一微秒的 CPU 时间都是有主人的，不能让调试工具（打印）抢了控制逻辑的资源。**
